# Wand PWA 转 Android APK 方案

## 结论先说

工作量不算特别大，但**不是“直接把 PWA 打包一下”就完事**。

更准确地说，这个方向应该定义为：

- `wand` 服务继续跑在服务器/电脑上
- Android APP 作为 **远程客户端** 去连接现有 `wand` HTTP/WebSocket 服务
- APP 负责连接配置、扫码导入、通知、移动端壳体验
- 现有 Web UI / REST API / WebSocket 尽量复用

如果目标只是：
- 在安卓上安装成一个 App 图标
- 打开后连接现有 wand 服务
- 基本复用现有前后端接口

那么最现实的路线是：**做一个 Android WebView 壳应用**，让 APK 内嵌浏览器访问 wand 服务。

粗分工作量：

| 目标 | 工作量 | 说明 |
|---|---:|---|
| 仅做“能安装的 APK 壳” | 1~2 天 | WebView + 基础权限 + 启动页 |
| 做到“可稳定日常使用” | 3~5 天 | 证书、网络发现、断线提示、前后台体验 |
| 做到“像原生 App 一样顺手” | 1~2 周 | 文件选择、通知、分享、深链、错误恢复、发布细节 |

更准确地说，**难点不在页面本身，而在移动端连接本地/局域网服务这件事**。

## 现在的接口能不能直接复用

### 结论

**大部分可以直接复用。**

你现在的 Web UI 本身就是浏览器端页面，后端已经提供了：
- REST API：`/api/*`
- WebSocket：`/ws`
- PWA Manifest：`/manifest.json`
- Service Worker：`/sw.js`

从代码看：
- PWA manifest 在 `src/pwa.ts:7`
- service worker 在 `src/pwa.ts:41`
- manifest 路由在 `src/server.ts:449`
- sw 路由在 `src/server.ts:466`
- WebSocket 在 `src/server.ts:979`
- 前端大量调用 `/api/*`，WebSocket 也已接入，见 `src/web-ui/content/scripts.js:831`、`src/web-ui/content/scripts.js:4352`、`src/web-ui/content/scripts.js:4487`

所以如果 APK 只是一个壳：
- **HTTP API 不用重写**
- **WebSocket 不用重写**
- **前端页面大概率也不用重写**

### 但不是“100% 无脑复用”

需要额外处理这几类问题：

1. **地址来源**
   - APK 打开时连哪个地址？
   - `127.0.0.1` 只代表手机自己，不代表运行 wand 的电脑。
   - 所以通常要改成：
     - 手动输入服务器地址
     - 扫码导入地址
     - 局域网发现

2. **HTTPS / 自签证书**
   - 你项目支持自签 HTTPS，见 `src/server.ts:385`。
   - 但 Android WebView 对**自签证书**通常更敏感。
   - 这会比普通浏览器里更麻烦。

3. **认证态 / Cookie / WebSocket**
   - 现有登录依赖 Cookie，见 `src/server.ts:261`、`src/server.ts:269`。
   - WebView 一般能带 Cookie，但要专门验证：
     - 登录后 API 是否持续带 cookie
     - WebSocket 握手是否沿用 cookie

4. **PWA 的一部分能力在 WebView 里未必等价**
   - Service Worker 在现代 Android WebView 通常可用，但兼容性和行为仍需实测。
   - 即使可用，也未必有“浏览器安装版 PWA”那种完整宿主能力。

## 推荐路线

## 方案 A：Android 原生壳 + WebView（推荐）

### 做法

新建一个很薄的 Android App：
- 主界面只有一个 `WebView`
- 首次启动配置 wand 服务地址
- App 内加载 `https://你的电脑IP:9443/` 或 `http://你的电脑IP:9443/`
- 保持现有前后端不变

### 优点

- 改动最小
- 最大化复用现有 UI / API / WS
- 上线和本地安装都简单
- 后续可逐步增加一点原生能力

### 缺点

- 依赖远端 wand 服务在线
- 网络、证书、局域网访问是主要坑
- 不是完全离线 App

### 适合你当前项目的原因

你这个项目本质上就是：
- Node.js 服务端
- 浏览器 UI
- WebSocket 实时交互

它天然就更适合“**原生壳包网页**”，而不是重写成纯安卓原生客户端。

---

## 方案 B：Capacitor 打包（可行，但不是最优）

### 做法

把前端抽成更标准的静态 Web App，再用 Capacitor 生成 Android 工程。

### 问题

你当前前端不是独立 SPA 构建产物，而是：
- 服务端 `renderApp()` 组 HTML，见 `src/web-ui/index.ts`
- 静态内容从 `src/web-ui/content/` 拷贝到 `dist/web-ui/`

这意味着如果走 Capacitor：
- 要先把前端从服务端耦合里拆出来
- 或者仍旧在 Capacitor WebView 里访问远端 wand 服务

这样一来，Capacitor 相比“原生 WebView 壳”优势不大，反而多一层工程复杂度。

### 结论

**可做，但不如方案 A 直接。**

---

## 方案 C：把 Node 服务也塞进 APK（不推荐）

### 做法

尝试在安卓端运行 Node.js / PTY / SQLite / 本地 CLI，做真正“端内完整 wand”。

### 问题

这会撞上几个大坑：
- Android 内嵌 Node 运行时复杂
- `node-pty` 在 Android 上几乎不是直接兼容路线
- Claude Code / 本地 CLI 的宿主环境、权限、TTY 语义都不一样
- 文件系统和进程管理会很麻烦

### 结论

对当前项目来说，这基本相当于**另起一个产品方向**，不建议。

## 需要额外做的事情

下面这些是 APK 化时要补的，不是现有 PWA 自动拥有的。

另外，建议把连接方式明确成两条主路径：
- **设置页生成连接信息**：在 wand 设置页展示二维码 + 连接串
- **APP 侧导入连接**：扫码、粘贴连接串、手动输入 URL

这样 APP 首版就不需要自己做复杂发现协议，连接闭环会更简单。

### 建议的连接串

建议统一成类似：

```text
wand://connect?url=https%3A%2F%2Fexample.com%3A9443&name=Home
```

这样可以同时用于：
- 设置页二维码
- 一键复制
- APP 深链打开
- 最近连接保存

如果 V1 想更简单，也可以先只传一个裸 URL，但长期看还是连接串更稳。

| 模块 | 是否必须 | 难度 | 说明 |
|---|---|---|---|
| Android 工程骨架 | 必须 | 低 | 建 WebView App、包名、签名配置 |
| 服务地址配置页 | 必须 | 低 | 首次输入 `host:port`，保存本地 |
| 局域网访问方案 | 必须 | 中 | 手机必须能访问运行 wand 的机器 |
| HTTP/HTTPS 策略 | 必须 | 中 | 明确只支持 HTTPS 还是允许局域网 HTTP |
| 自签证书处理 | 高优先 | 高 | 这是最容易卡的点 |
| WebView Cookie 验证 | 必须 | 中 | 登录态、会话恢复、接口调用要通 |
| WebSocket 连通性验证 | 必须 | 中 | `/ws` 是否稳定重连 |
| 离线/断线页 | 建议 | 低 | 服务不可达时给出明确提示 |
| 安卓返回键行为 | 建议 | 低 | WebView 页面返回 vs 退出 App |
| 文件上传/下载适配 | 看需求 | 中 | 如果未来加附件/导出，会涉及原生桥 |
| 通知 | 可选 | 中 | 会话完成、权限等待可做推送/本地通知 |
| 二维码配对 | 建议 | 中 | 降低手输地址成本 |
| 图标/启动页/应用名 | 必须 | 低 | APK 基础包装 |
| 发布签名/安装包流程 | 必须 | 低 | debug/release keystore |

## 接口复用评估

| 能力 | 现状 | APK 中是否可复用 | 备注 |
|---|---|---|---|
| 页面 UI | 已有服务端渲染页面 | 可复用 | 直接 WebView 打开即可 |
| REST API `/api/*` | 已有 | 可复用 | 关键是地址与认证 |
| WebSocket `/ws` | 已有 | 可复用 | 需验证 Cookie 和断线重连 |
| 登录认证 | Cookie Session | 基本可复用 | 需验证 WebView Cookie 策略 |
| PWA manifest | 已有 | 不重要 | APK 场景下不是核心 |
| Service Worker | 已有 | 部分可复用 | 不是 APK 成败关键 |
| 本地安装能力 | PWA 已支持 | 不复用 | APK 安装走 Android 包机制 |
| 推送/通知 | 基本没有原生能力 | 不能直接复用 | 需要 Android 原生补 |
| 原生分享/扫码/文件权限 | 现有没有 | 不能直接复用 | 需原生能力接入 |

结论：

**业务接口层基本能复用，宿主能力层不能直接复用。**

## 最大的几个难点

## 1. 自签 HTTPS

这是第一难点。

你现在支持：
- `https: true`
- 自签证书生成

但在 Android APK 里：
- 如果用 WebView 打开自签 HTTPS 地址
- 很可能遇到证书信任问题
- 即使普通浏览器能手动点过去，WebView 里策略也未必一样

### 处理办法

有三种路：

| 方案 | 难度 | 适合度 | 说明 |
|---|---:|---:|---|
| 局域网内先走 HTTP | 低 | 高 | 开发最快，但安全性弱一些 |
| 给 wand 配可信证书 | 中高 | 中 | 局域网/内网环境下不一定方便 |
| Android 里做自签信任放行 | 高 | 低 | 能做，但维护成本高，不建议优先 |

**建议顺序：先支持局域网 HTTP 开发验证，再决定 HTTPS 正式方案。**

---

## 2. 手机如何找到你的 wand 服务

PWA 在浏览器里你可以自己访问一个 URL。
但 APK 面向日常使用时，不能要求每次都手打地址。

建议至少做一个：
- 手输地址
- 最近连接记录
- 二维码扫码导入

推荐组合：
- **V1：手输地址 + 最近连接**
- **V2：加二维码配对**

---

## 3. WebSocket 稳定性

wand 很依赖实时流：
- 会话输出
- 权限提示
- 状态同步

所以 APK 化后，必须重点验证：
- 切后台再回来，WS 是否自动恢复
- 手机锁屏后再解锁，是否能恢复
- 网络从 Wi‑Fi 切换到蜂窝/反过来，是否能重连

这部分后端接口大概率不用改，但**前端重连策略可能要增强**。

---

## 4. 触屏体验不是完全等于桌面浏览器

虽然你已有移动端优化，但 APK 化后用户预期更像原生 App。

常见还要补：
- 返回键逻辑
- 输入框聚焦与键盘弹出
- 横竖屏切换
- WebView 下载/打开外链行为
- 证书错误页的可解释性

这些通常不是大重构，但会有不少边角处理。

## 一个实际可落地的分阶段方案

## Phase 0：可行性验证

目标：确认“复用现有接口 + WebView 壳”能跑通。

要做：
1. 建一个最小 Android WebView App
2. 手动输入 wand 地址
3. 打开首页
4. 验证登录
5. 验证 `/api/config`、`/api/sessions`
6. 验证 `/ws` 订阅与消息流
7. 验证创建会话、发送消息、恢复会话

产出：
- 一份“能不能通”的验证结论
- 一份问题清单（证书 / Cookie / WS / UI）

### 这阶段工作量

**小。**
主要是验证，不是开发大功能。

---

## Phase 1：做成能安装能用的 APK

目标：本地安装后，能稳定连接并使用 wand。

要做：
1. App 首页做服务器地址配置
2. 保存最近连接地址
3. WebView 加载远端地址
4. 处理断网/服务不可达提示
5. 处理返回键
6. 基础图标、启动页、应用名
7. Debug/Release 签名打包

### 这阶段工作量

**中等。**
是最值得做的一期。

---

## Phase 2：补移动端体验

目标：从“能用”提升到“顺手”。

要做：
1. 二维码扫码导入服务器地址
2. 更好的重连提示与状态展示
3. 后台恢复策略
4. 下载/外链处理
5. 可选通知能力
6. 针对 WebView 的 UI 小修

### 这阶段工作量

**中等偏上。**
不是必须，但会明显影响观感。

## 对现有代码可能需要的小改动

虽然原则上接口能复用，但建议预留几项小改动位：

| 改动点 | 必要性 | 说明 |
|---|---|---|
| 增加“服务健康检查”接口 | 建议 | 例如返回版本、协议、是否需登录，方便 APK 启动页探测 |
| 明确 WebSocket 断线状态 | 建议 | 方便移动端做更明确提示 |
| 登录态接口更清晰 | 建议 | 方便 App 启动时判断“需登录 / 已登录” |
| 允许更友好的证书/网络错误反馈 | 建议 | 方便前端展示移动端引导 |
| 提供二维码/连接串配对页 | 建议 | 方便把服务器地址发给手机 |
| 提供 App deep link 兼容的连接串 | 建议 | 方便 `wand://connect?...` 直接导入 |

注意：
这些都属于**增强型改动**，不是必须先做。

## 仓库组织方案

你提的这个点很关键：如果后面 Web 端和 APP 端要经常一起改，目录组织会直接影响协作效率。

这里主要有三种组织方式。

### 方案 1：在当前仓库内直接新增 `android/` 目录（最推荐）

示意：

```text
wand/
  src/
  dist/
  android/
    app/
    gradle/
    ...
```

#### 优点
- Web 端和 APP 端放在一个仓库里，联动修改最方便
- 一个 issue / 一个分支里可以同时改服务端、Web UI、Android
- 不需要处理 submodule 同步问题
- 代码评审和发布记录集中

#### 缺点
- 仓库会变大一些
- Android 工程文件会混入当前项目
- Node/Android 两套开发环境会共存

#### 适用判断
如果你的预期是：
- 后面会频繁做“服务端 + Web + APP”联动改动
- 并且 APP 本质上就是 wand 的官方客户端

那这个是**最顺手**的结构。

### 方案 2：单独一个 Android 仓库，用 submodule 挂进来（不太推荐）

示意：

```text
wand/
  src/
  android-app/   # git submodule
```

#### 优点
- Android 工程边界很清晰
- 可以独立发版、独立权限控制
- Android 团队可完全独立节奏

#### 缺点
- submodule 日常维护麻烦
- 联动改动时要跨两个仓库提交
- 新人容易在 submodule 上踩坑
- 版本对应关系要额外维护

#### 适用判断
只有在下面情况才更合理：
- APP 团队独立明显
- 发版节奏独立明显
- 权限/仓库管理必须拆开

对于你现在这个阶段，**太重了**。

### 方案 3：在上层目录做一个聚合仓，`wand` 和 `wand-android` 并列（次优）

示意：

```text
workspace/
  wand/
  wand-android/
```

或者做一个上层聚合仓：

```text
wand-suite/
  wand/          # 子目录或子仓
  wand-android/  # 子目录或子仓
```

#### 优点
- Web/服务端 与 Android 边界清楚
- 不必用 submodule 也能分仓管理
- 两边都比较独立

#### 缺点
- 联动改动仍然不是一个仓内完成
- issue、版本、分支协作没单仓顺
- 如果要共享文档/脚本，维护会分散

#### 适用判断
如果你明确想让 Android 未来可能独立，但眼下又不想碰 submodule，这个方案可以接受。

## 目录组织推荐结论

### 当前最推荐

**直接在当前仓库里新增 `android/` 目录。**

原因很简单：
- APP 不是一个完全独立产品，而是 wand 的客户端
- 你已经预期后面会有 Web 端和 APP 端的联动改动
- 那么单仓协作成本最低

建议结构：

```text
wand/
  src/
  dist/
  android/
  android-docs/
```

如果不想多一个文档目录，也可以直接：

```text
wand/
  src/
  android/
  docs/
```

或者更轻一点：

```text
wand/
  src/
  android/
  android-apk-plan.md
```

### 不建议当前就上 submodule

原因：
- 现在还在方案验证期
- Android 端和服务端/API 会频繁对齐
- submodule 会放大管理成本，不会带来实质收益

### 什么时候再考虑拆仓 / submodule

只有当下面条件成立时再考虑：
- Android 已经稳定成独立产品线
- 发版频率明显独立
- 团队分工明显独立
- API 边界已经非常稳定

在那之前，**先单仓最合适**。

- 改 Web 连接页

- 改 Android 扫码导入
- 一个仓库一个分支一起提交

## 对目录组织的最终建议

| 方案 | 推荐度 | 适合当前阶段吗 | 说明 |
|---|---|---|---|
| 当前仓库新增 `android/` | 高 | 是 | 最适合联动开发 |

## 我建议的最终技术决策

### 最推荐

**Android 原生 WebView 壳 + 继续复用现有 wand 服务端与前端。**

### 不建议一开始做

- 不建议先重构成 Capacitor 主导架构
- 不建议尝试把 Node/PTy/Claude 全塞进 APK
- 不建议一开始就深挖 WebView 自签证书特判

### 最务实的优先级

1. 先做 WebView 壳 PoC
2. 先用局域网 HTTP 或可控 HTTPS 验证功能
3. 跑通登录/API/WS
4. 再补连接配置、二维码、断线恢复
5. 最后再决定 HTTPS 正式方案

| WebView 壳工程 | 低 | 不麻烦 | 标准 Android 活 |
| 现有页面复用 | 低 | 不麻烦 | 直接加载即可 |
| REST API 复用 | 低 | 不麻烦 | 基本直接用 |
| WebSocket 复用 | 中 | 有点麻烦 | 主要看移动网络切换 |
| Cookie 登录态 | 中 | 有点麻烦 | 需实测 WebView 行为 |
| 服务地址配置 | 低 | 不麻烦 | UI 小功能 |
| 二维码配对 | 中 | 一般 | 提升体验明显 |
| 断线/重连体验 | 中 | 一般 | 移动端必做优化 |
| 自签 HTTPS | 高 | 很麻烦 | 最大不确定点 |
| 原生通知/分享/文件能力 | 中 | 一般 | 属于增强项 |
| 把 Node/PTY 直接塞 APK | 很高 | 非常麻烦 | 不建议 |

## 最短回答版

如果你只是问：

### “现在这套 PWA 做成安卓 APK，难不难？”

答案是：

**中等，不算特别大。**

### “接口能不能复用？”

答案是：

**绝大多数能复用。REST 和 WebSocket 基本都可以沿用。**

### “真正麻烦的点是什么？”

答案是：

**不是业务接口，而是移动端宿主问题：服务器地址发现、WebView Cookie、断线重连、尤其是自签 HTTPS。**

### “推荐怎么做？”

答案是：

**先做一个 Android WebView 壳 App，当远端客户端来连现有 wand 服务，这是当前成本最低、成功率最高的路线。**
